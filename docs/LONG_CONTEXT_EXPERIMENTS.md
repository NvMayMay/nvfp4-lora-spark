# Long-context Super training experiments

> Cross-model context-fit tracking (all families under the unified trainer, incl. FLCE
> conditions) lives in [CONTEXT_FIT_MATRIX.md](CONTEXT_FIT_MATRIX.md). This doc is the
> Super-120B (legacy `train/train_super_nvfp4.py`) deep dive.

This log tracks bounded experiments to increase the Super-120B NVFP4 LoRA training context window. The standing rule is to use small smoke runs first, normally `--limit-examples 50`, and avoid full-dataset training unless explicitly promoted later.

## Operating rules

- Focus experiments on `train/train_super_nvfp4.py`; Nano may inherit code improvements but is not the current bottleneck.
- Every attempted run gets an entry with command, outcome, memory/step notes, and the next decision.
- Prefer accumulation-boundary stops/checkpoints so resume semantics match uninterrupted training.
- Use `--mask-prompt-labels` for new training-quality experiments.
- Legacy sweep reproduction requires `--pad-to-max-length --no-length-bucketing --grad-accum 1`.

## Design decisions

### 2026-05-27: Dynamic padding and bucketing are default for batched Super smokes

Batched training previously padded every sample to global `--max-len`, which makes long-context experiments pay for padding on short examples. New batched runs default to padding only to the longest item in the batch and sorting records by tokenized length. This changes data order for batched runs, so legacy checkpoints and sweep reproduction must use the explicit legacy flags.

### 2026-05-27: Smoke tests use dataset limiting, not just early stop

Added `--limit-examples` so smoke tests operate on a bounded slice before batching. This keeps exploratory runs from accidentally rendering/tokenizing or training over the full dataset.

### 2026-05-27: Checkpoints and stops should align with accumulation boundaries

For `--grad-accum > 1`, saving after a micro-step but before `optimizer.step()` drops pending `.grad` buffers on resume. The scripts now only periodic-checkpoint on accumulation boundaries and only honor `--stop-at-step` once the current accumulation window has completed.

### 2026-05-27: Super smoke runs use CLI path overrides and skip final adapter saves

Added `--model-dir`, `--train-file`, and `--adapter-dir` to the Super training script so experiments can target the real local model/data without editing script constants. Added `--no-save-final-adapter` for crash/fit smokes, because writing a full Super adapter at the end of every 50-example probe is slow and wastes disk. Periodic checkpoints still occur on accumulation boundaries if a run reaches `SAVE_EVERY`.

### 2026-05-27: Immediate research backlog from sidecar agent

Harvey ranked the next practical Super context levers as: deeper activation-checkpoint audit and `config.use_cache=False`, 8-bit/paged optimizer states, fused or chunked linear cross entropy to avoid full fp32 logits, LoRA rank/target ablations, attention backend checks, and `save_on_cpu` as an experimental offload. First local tests will use the lowest-risk controls already added; optimizer/logit-loss changes require separate code patches after baseline Super 4096 behavior is known.

### 2026-05-27: Stop co-tenant Qwen serve before Super experiments

`nvidia-smi` showed the existing `qwen_ich_v34_serve` tmux process holding about 68.2 GiB of GPU memory. Super-120B NVFP4 training cannot load reliably with only ~55 GiB CUDA free. For this Super-only context investigation, the Qwen serve process is treated as a co-tenant and should be stopped before retrying Super smokes.

### 2026-05-27: Qwen serve stopped for exclusive Super GPU experiments

After full access was granted, stopped tmux session `qwen_ich_v34_serve`. `nvidia-smi` then showed only desktop GPU processes and `free -h` showed about 117 GiB available system memory. Super experiments can now run single-tenant, matching the repo's operational guidance.

### 2026-05-27: Need long-example selection for real context pressure

`--max-len 4096` alone does not guarantee a 4096-token activation test. The first-50 smoke slice logged `seq_len=497` on step 1. Next Super-only patch should add a deterministic `--select-longest-examples` or equivalent after tokenization so bounded smokes can deliberately choose the longest samples without using the full dataset for training.

### 2026-05-27: Corpus max is 2577 tokens under 4096 cap

Selecting the 50 longest examples from all 1081 training rows showed max tokenized length 2577 at `--max-len 4096`. Future context-window fit tests need synthetic or alternate long examples; otherwise increasing `--max-len` mostly tests config plumbing, not activation memory at the target length.

### 2026-05-27: Synthetic pressure records are fit tests, not quality tests

The synthetic repeat mode deliberately repeats a real seed record to create exact-length token-ID samples. Use these runs for crash/fit mapping only. They do not validate data distribution, training quality, or general long-context behavior.

### 2026-05-27: Do not treat low-rank or tiny-target probes as a fix

Reducing LoRA rank or dropping target modules is useful for mapping where activation and allocator pressure break, but it is not a satisfying context-window solution. The current useful Super frontier remains 28672 tokens with the normal r=8 `up_proj,down_proj` adapter. The 32768 result is a reduced-rank r=4 compromise. The 34816 result uses only `down_proj` at r=2 and should be treated as a tiny-adapter boundary probe, not a training recipe.

### 2026-05-27: 36864 tiny-adapter probe caused host instability

`SUPER-LC-024` loaded successfully and entered the first 36864-token training step, but no Python exception or clean CUDA OOM was written to the run log. Kernel logs from the previous boot show a flood of NVIDIA `NV_ERR_NO_MEMORY` failures from `_memdescAllocInternal(pMemDesc) @ mem_desc.c:1359`, ending at 20:37:38. System services then started timing out, and the machine was later shut down via the power key at 20:46:57. This looks like NVIDIA kernel/allocator exhaustion under unified-memory pressure rather than a catchable PyTorch failure. After reboot the system is on `6.17.0-1018-nvidia` with matching NVIDIA `580.159.03`; the failed run happened on the older `6.17.0-1014-nvidia` boot with NVIDIA module `580.142`.

Decision: do not continue by shrinking rank/targets further or by retrying 36864 immediately. Resume from known-good r=8 or r=4 configurations and add a hard safety margin, host-memory watchdog, and activation-memory instrumentation before any higher-context probe.

## Post-crash investigation plan

### Safety gate

All new Super frontier/profiling runs should include:

`--watchdog-min-available-gb 2 --watchdog-nvrm-errors --profile-memory-phases`

The RAM threshold is intentionally low because the user asked for a 2 GiB floor, but it is still enough to abort before the machine spends minutes with almost no reclaimable host memory. The NVRM watcher tails new kernel log lines and aborts on fresh `NV_ERR_NO_MEMORY` / `mem_desc.c:1359` allocator errors.

Watchdog self-tests passed on 2026-05-27 without loading the model: an impossible RAM threshold exited via code 90, and the NVRM journal watcher started/stopped cleanly.

### Evidence-gathering sequence

1. Validate the watchdog on a safe short/load-only Super path.
2. Re-run a known-good r=8 configuration at 24576 or 28672 with watchdog enabled to confirm the new kernel/driver stack is stable and comparable.
3. Use `--profile-memory-phases` on r=8, not low-rank probes. Capture per-step CUDA peak/reserved, host `MemAvailable`, and whether peak occurs in forward, backward, loss, or optimizer.
4. Use the profile to choose the next real lever: deeper checkpoint coverage, saved-tensor/offload hooks, attention/Mamba activation reductions, or a loss/backward memory reduction.
5. Only retry a frontier context when a previous known-good r=8 run shows several GiB of recovered headroom. Do not promote r=2/down-only results as context-window fixes.

### 2026-05-27: First watchdog-profiled 24576 r=8 rerun hit NVRM allocator errors

`SUPER-LC-025` showed that the watchdog is not just theoretical. The model loaded successfully with pooled buffers, but the first 24576-token r=8 training step produced fresh NVIDIA `NV_ERR_NO_MEMORY` / `mem_desc.c:1359` errors while host `MemAvailable` was still about 39 GiB. The watchdog exited with code 90 and the host recovered cleanly. Treat this as `UNSAFE/controlled-abort`, not a memory-capacity pass.

Decision: before any more 24k+ r=8 probes, attack the likely transient allocator source flagged by code inspection: the remote MoE implementation builds a dense `one_hot(topk_indices, num_classes=512)` tensor for every MoE layer. Add an opt-in sparse MoE routing patch, validate it at shorter context, then re-profile.

### 2026-05-27: MoE sparse no-one-hot patch

Added opt-in `--moe-sparse-no-one-hot` in `train/train_super_nvfp4.py`. It monkey-patches loaded `NemotronHMoE` modules so the expert loop uses `torch.where(topk_indices == expert_idx)` directly instead of materializing the dense `[tokens, top_k, n_experts]` one-hot routing mask. A tiny fake-MoE equivalence test matched the original dense algorithm exactly: output max abs diff 0, hidden-gradient max abs diff 0, routing-weight-gradient max abs diff 0.

### 2026-05-27: 8192 r=8 with MoE sparse still hit NVRM during backward

`SUPER-LC-026` validated that the MoE sparse patch reduces early forward memory, but it did not eliminate the NVIDIA allocator failure under the strict watchdog. The run reached `after_chunked_loss` at 8192 tokens with `cuda_peak=88.56GB` and `cuda_reserved=88.60GB`, lower than the earlier pooled 8192 run's final `cuda_peak=92.28GB` / `cuda_reserved=94.97GB`. The watchdog then caught fresh `NV_ERR_NO_MEMORY` / `mem_desc.c:1359` before `after_backward`.

Decision: stop GPU experiments in this boot/session. After the second controlled NVRM abort, a fresh Python process reported only `97.5GB` CUDA free out of `130.7GB` while `nvidia-smi` showed no compute process, suggesting driver-side residue. Reboot or otherwise reset the GPU/driver before further Super training tests. Next software step should investigate backward-time allocation pressure and/or driver allocation-count pressure, not higher context.

### 2026-05-27: Post-reboot strategic pivot

The current full-sequence r=8 Super path appears to have a large fixed memory floor plus a backward-time allocator cliff:

- Pooled model load alone allocates about 81.7GB CUDA.
- Full r=8 `up_proj,down_proj` LoRA trains about 1.216B parameters across 81922 tensors, so gradients and optimizer metadata are meaningful even with Adafactor.
- Gradient checkpointing is already enabled at block granularity, chunked LM-head CE avoids full-sequence logits, SDPA no-mask avoids the explicit attention causal mask for unpadded runs, and pooled loader buffers reduce load time/allocation churn.
- The remaining failure mode is not simply raw host RAM exhaustion: the watchdog observed fresh NVIDIA `NV_ERR_NO_MEMORY` allocator errors while host `MemAvailable` was still about 39GiB on one run. That points toward CUDA/NVIDIA allocation pressure during backward recompute and/or large transient workspaces.

Decision: do not continue treating low rank or tiny target sets as the main path. The serious r=8 full-sequence frontier remains 28672 tokens as a crash/fit result, but under the 2GiB watchdog it is not safety-certified. The next high-value direction is a cached prefix / trainable suffix mode: process long prefixes under `torch.no_grad()` into Nemotron-H hybrid cache, then backprop only through a suffix block. This is approximate truncated-BPTT training, not exact full-sequence training, but it can expose the model to much longer effective context without scaling activation memory with the entire sequence.

Target strategy:

1. Re-establish a post-reboot safety baseline at short context with the watchdog enabled.
2. Add or prototype cached-prefix/suffix-loss training for Super. Keep r=8 and the normal target modules.
3. Smoke test effective contexts such as 32768, 65536, and 131072 with trainable suffix windows of 4096-16384 tokens.
4. Separately continue exact full-sequence memory work: router detachment experiment, Mamba chunk-size sweep, finer checkpoint granularity, and allocator-count reduction.
5. Treat 262144 as the current declared model position ceiling. The user's 500k goal requires a separate position-extension decision after memory strategy is solved.

Added `--synthetic-repeat-to-len` for Super-only context smokes. It repeats token IDs from the longest available example to force an exact sequence length, so it is useful for activation-memory and crash testing but not for evaluating loss quality or instruction-tuning behavior. Synthetic pressure runs require `--batch 1` and omit `--mask-prompt-labels`.

### 2026-05-27: Next memory levers after synthetic frontier

Sidecar static audit ranked the next Super-local memory wins as chunked/fused cross entropy first, because the model's HF loss path materializes full fp32 logits and shifted-logit copies, then 8-bit or low-state optimizer modes, then attention backend experiments. Once the raw synthetic frontier is known, prioritize a `--loss-mode chunked_ce` patch before broader context sweeps, because logits scale directly with sequence length and vocab size.

### 2026-05-27: Chunked frozen-LM-head CE added as an experimental loss mode

Added `nvfp4_lora/loss.py` and `--loss-mode chunked_frozen_ce --loss-chunk-tokens N` for Super training. This path calls the base model for hidden states, then computes cross entropy through the frozen LM head in token chunks using a custom autograd function. It avoids retaining full-sequence fp32 logits for backward. A toy CUDA comparison against standard CE matched loss within `2.4e-7` and hidden gradients exactly; the first Super use should still be treated as experimental until compared on a short real sequence.

### 2026-05-28: Loss chunk size is an allocator-pressure knob

At 12288 tokens, full-rank `--loss-chunk-tokens 128` failed under the NVRM watchdog immediately after chunked loss, but the same run with `--loss-chunk-tokens 512` passed through backward and optimizer. Forward/loss peak memory was essentially unchanged, so the improvement is likely fewer repeated LM-head/logit allocation cycles rather than lower peak tensor memory. New long-context exact-training smokes should use 512 unless a later test proves a better value.

### 2026-05-28: 20k exact full-rank remains unsafe under watchdog

Multiple 20480-token full-rank attempts failed with fresh `NV_ERR_NO_MEMORY` / `mem_desc.c:1359` before backward despite 21GiB+ host `MemAvailable` after forward. `cudaMallocAsync` failed earlier than the native allocator. Avoiding expanded scale scratch in `dequantize_nvfp4_weight` was correct and equivalence-tested, but did not move the 20k boundary. Bf16 logits for chunked CE were mechanically valid in a toy gradient check, but also failed earlier at 20k. Current exact, full-rank, watchdog-certified ceiling is therefore 16384 tokens.

### 2026-05-28: Cached-prefix training fixed with readonly cache and suffix checkpointing

The first cached-prefix prototype failed even at a 512-token train suffix because cached-prefix mode disabled gradient checkpointing and mutated the prefix cache during the trainable suffix forward. The fix is to prefill the prefix under `torch.no_grad()` with checkpointing temporarily disabled, then make the cache read-only for the suffix so gradient checkpointing can remain enabled during suffix training. Attention layers concatenate detached prefix K/V with trainable suffix K/V without writing back into the cache; Mamba cached continuation reads prefix conv/SSM states as initial states without writing final suffix states back while the cache is marked read-only.

This changes cached-prefix training into an approximate objective: only suffix tokens receive gradients, and cached-prefix compare is not exact-equivalent at hidden-state level. It is therefore a long-context adaptation mode, not a drop-in replacement for exact full-sequence training.

### 2026-05-28: Cached suffix attention uses lower-right causal bias

For cached suffixes, `q_len != kv_len`, and plain PyTorch `is_causal=True` uses upper-left causal alignment. That is wrong for prefix-cache continuation. The SDPA no-mask patch now keeps full-sequence `is_causal=True` only when `cache_position[0] == 0`; cached suffixes use `torch.nn.attention.bias.causal_lower_right(q_len, kv_len)` instead. A local SDPA test matched this bias exactly against the dense lower-right mask (`max_diff=0.0`) for mismatched query/key lengths. This avoids materializing a dense `(suffix_len, total_len)` causal mask before larger-prefix experiments.

### 2026-05-28: Static prefix K/V cache added

Prefix prefill initially used the remote dynamic cache, whose attention `update` grows K/V with `torch.cat` on every prefix chunk. That fit through 64k but made prefill wall time the dominant bottleneck and caused avoidable allocation churn. Cached-prefix prefill now preallocates each attention layer's prefix K/V tensor to the final prefix length on first use, writes each chunk into a slice, and returns a view of the filled prefix so far. This keeps the suffix readonly-cache behavior unchanged, but avoids repeated full-prefix K/V reallocations during prefill.

### 2026-05-28: Inactive MoE dummy expert calls removed

The sparse MoE patch originally avoided the one-hot router tensor but still called every inactive expert on a dummy token. That kept enormous useless LoRA/dequant graph surface alive in MoE layers. The inactive branch now does no work: experts with no routed tokens contribute zero output and receive no gradient for that step, which is the natural sparse-MoE behavior.

### 2026-05-28: Sparse MoE now visits only routed experts

The sparse MoE patch was tightened again so each MoE layer builds a sorted flat route list and visits only experts present in `topk_indices`, rather than scanning every expert with an empty `torch.where`. This does not lower LoRA rank or target capacity; it only prevents inactive experts from entering the suffix trainable path and reduces temporary routing allocations. A CPU fake-MoE equivalence/gradient smoke matched the reference loop exactly (`max_abs_diff=0.0`).

### 2026-05-28: Cached suffix SDPA keeps native GQA

Cached suffixes need a lower-right causal bias because `q_len != kv_len`. A research pass found that sending that non-`None` bias through the Transformers SDPA wrapper can expand grouped-query attention K/V heads before SDPA. The attention patch now calls `torch.nn.functional.scaled_dot_product_attention(..., enable_gqa=True)` directly for cached suffixes with lower-right bias, keeping compact K/V representation while preserving causal alignment. A CPU API smoke verified the local PyTorch build accepts `causal_lower_right` with `enable_gqa=True`.

### 2026-05-28: Readonly cached suffix no longer asks Mamba for final states

The cached Mamba continuation patch previously requested final conv/SSM states even when the suffix cache was read-only and those states would be discarded. In readonly suffix mode it now sets `return_final_states=False` for both causal conv and chunk scan. This should trim suffix forward/backward allocator pressure without changing rank, target modules, or the trainable-token objective.

### 2026-05-28: Activation offload and Mamba chunk-size knobs added

Added `--activation-offload none|save_on_cpu`. The `save_on_cpu` mode wraps trainable forward/loss regions with `torch.autograd.graph.save_on_cpu(pin_memory=True)` so tensors saved for backward leave the CUDA caching allocator. This is aimed at the NVRM descriptor-pool failure signature rather than raw byte pressure and must be validated because it may increase host/pinned-memory churn.

Added `--mamba-chunk-size N` to override loaded Nemotron-H Mamba scan chunk sizes. Default remains the checkpoint value. Smaller chunks are an allocator-shape experiment, not an assumed win; too small may increase launch/allocation churn.

### 2026-05-28: External feedback triage and priority order

I buy the descriptor-table diagnosis. It explains the observed pattern better than plain "we ran out of GB": similar peaks behave differently, `cudaMallocAsync` failed earlier, and the kernel failures consistently point at NVRM `mem_desc.c:1359`. The priority is therefore reducing the count and lifetime of distinct allocations crossing the backward boundary, not only shrinking tensor bytes.

Priority read:

- Immediate P0: direct native-GQA SDPA for cached suffix, sorted sparse MoE, and readonly Mamba final-state suppression. These are patched and validated at 4096/2048 in SUPER-LC-059.
- Next P0/P1: `save_on_cpu` as an opt-in. This is a strong theoretical fit on UMA because it may release CUDA allocator VA segments/descriptors, but it must stay behind a flag and be tested first on known-good rungs because it can introduce pinned-memory churn and wall-time surprises.
- Good but not first: Liger FLCE. The critique of current chunked CE scratch is fair, but `liger_kernel` is not installed in the current venv, so this should be optional after install/API validation rather than silently vendored into the training path.
- Useful ablation, not free: LoRA-FA. It preserves rank, but freezing A still changes adapter capacity and optimization behavior. It is less destructive than dropping to r=2, but still a real training compromise.
- Mamba chunk size: worth exposing as a sweep knob. It is not assumed to help until tested because smaller chunks may reduce descriptor pressure or may simply increase launch/allocation churn.
- PoSE / Activation Beacon: strategically interesting but different objectives. Continue pushing the mechanical training ceiling first; use PoSE if the product goal is inference extrapolation, and consider Beacon if cached-prefix/suffix underfits long-range signal.

The external proposal's experiment numbering collided with the existing log. Continue local numbering from SUPER-LC-060 onward.

### 2026-05-27: SDPA no-mask patch added as an opt-in attention experiment

Added `--sdpa-causal-no-mask` for unpadded Super training. The remote Nemotron-H code defaults to SDPA, but still builds an explicit 4D causal mask unless this opt-in patch is used. The flag omits all-ones attention masks and monkey-patches attention calls so torch SDPA can use `is_causal=True`. Padded batches keep the original mask path. A tiny CUDA Nemotron-H attention equivalence test matched the original all-ones-mask output exactly (`max_abs_diff=0.0`), so the first real validation is a 24576-token synthetic Super rerun against SUPER-LC-013.

Added `--pooled-loader-buffers` and `--stop-after-load` to start testing loader-side allocation coalescing. The first pooled load-only probe exposed a bad metadata implementation: it was spending minutes in per-tensor safetensors metadata opens before reading meaningful checkpoint bytes. The metadata pass was changed to open each safetensors shard once; the pooled copy pass still needs validation and may need shard-grouped tensor reads next.

### 2026-05-27: LoRA rank is now configurable for fit probes

### 2026-05-27: LoRA target suffixes are configurable for frontier probes

Added `--target-suffixes`, defaulting to `up_proj,down_proj`. This allows memory/capacity ablations such as `--target-suffixes down_proj` to cut the number of trainable LoRA modules. Like rank reduction, this is a quality tradeoff for fit testing and must be evaluated before production use.
